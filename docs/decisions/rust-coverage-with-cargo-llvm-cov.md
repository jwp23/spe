# Rust Coverage with cargo-llvm-cov for SonarQube

Decision: Generate Rust code coverage in CI with `cargo-llvm-cov` (LCOV output) and upload it to SonarQube Cloud via the `sonar.rust.lcov.reportPaths` analysis parameter. The `SonarQube` workflow installs the `llvm-tools-preview` component, runs `cargo llvm-cov --lcov --output-path lcov.info -- --include-ignored` (covering both unit and `#[ignore]`d integration tests), and the scan reads `lcov.info` from the repo root.

Rationale: SonarQube Cloud's default "Sonar way" quality gate requires ≥80% coverage on new code, but CI uploaded no coverage report, so every `main` branch analysis scored 0% and the gate stayed red (`sonar.qualitygate.wait=true` then failed the scan). `cargo-llvm-cov` is the de facto Rust coverage tool, authored by the same maintainer as the related tooling, and emits LCOV — one of the two formats SonarQube's first-party Rust analyzer ingests. Including ignored integration tests in the coverage run yields an accurate project-wide number (~84% line coverage) rather than the unit-test-only subset. The tool is installed via `cargo install cargo-llvm-cov --version 0.8.7 --locked` rather than adding another third-party action to SHA-pin. The version is pinned — unlike the `audit` job's unpinned `cargo-audit` — because this tool's LCOV output feeds the quality gate: a format or behavior change in a future release could silently zero out coverage and turn the gate red, so reproducibility outweighs auto-currency here. Bump the pin manually; Dependabot's cargo ecosystem tracks `Cargo.toml`/lock, not `cargo install` version strings in a workflow.

Coverage matching depends on the scan running in the same checkout as the coverage step: `cargo-llvm-cov` writes absolute `SF:` paths, which resolve only because the scan action runs directly on the runner (not in a container) with `projectBaseDir` defaulting to the same `$GITHUB_WORKSPACE`. Moving the scan into a container or changing `projectBaseDir` would silently break file matching and zero out coverage.

Trade-off: The coverage step compiles `cargo-llvm-cov` and rebuilds the crate under instrumentation, adding wall-clock time to the `SonarQube` job (mitigated by `Swatinem/rust-cache`). Alternatives considered: relaxing the SonarCloud gate (loses the coverage signal) or setting `sonar.qualitygate.wait=false` (masks a red gate). Both were rejected in favor of making the gate measure real coverage. Tracked in spe-109.
