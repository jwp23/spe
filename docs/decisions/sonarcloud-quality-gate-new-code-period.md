# SonarCloud Quality Gate and New Code Period

Decision: Keep the built-in "Sonar way" (Clean-as-You-Code) quality gate and define the project's **New Code Period** as **"Previous version"**. Together with CI uploading coverage (see [rust-coverage-with-cargo-llvm-cov](rust-coverage-with-cargo-llvm-cov.md)), this makes the gate computable and meaningful for a project in maintenance mode.

Rationale: The `SonarQube` workflow failed on every `main` push because `sonar.qualitygate.wait=true` exits non-zero whenever the gate is anything other than passing. The real cause was *not* low coverage — the "Sonar way" gate has only new-code conditions, and the project had **no New Code Period defined**, so SonarCloud could not anchor "new code" and reported the gate as **"not computed" (NONE)**, which the scanner treats as a failure. (The original spe-109 diagnosis — "new-code coverage is 0%" — was wrong: `new_coverage` was never computed, not zero.) The project is not actively developed, so months may pass with no new commits. Setting New Code to "Previous version" while no `sonar.projectVersion` is ever set means the whole codebase counts as new code on each analysis, so the gate evaluates the real, now-uploaded overall coverage (84%) and all other conditions across the entire codebase — effectively gating on overall code, which is what a dormant project wants.

Trade-off: A custom quality gate with explicit overall-code conditions would have been the most direct expression of "review all code," but custom gates require a paid SonarCloud plan, so it is not available here. "Previous version" achieves the same effect on the free plan by exploiting the absence of version boundaries. Caveat: if `sonar.projectVersion` ever starts being set and bumped, "Previous version" would re-scope new code to since-the-last-version and could shrink it toward empty, risking a return of the "not computed" state — revisit the New Code Period if versioning is introduced. Tracked in spe-109.
